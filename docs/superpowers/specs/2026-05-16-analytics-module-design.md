# Analytics Module — Design Spec

## Overview

Add a conversation analytics module to the chatbot platform. The module provides two main pages:

1. **Sessions page** — browsable list of all conversations (active + completed) with filters and drill-down to individual session detail
2. **Analytics dashboard** — KPI cards, sentiment charts, resolution charts, trends, top queries, and NPS metrics

Analytics data is generated by a background worker job that runs a single Bedrock LLM call on each completed conversation, extracting sentiment, resolution status, satisfaction metrics, and a summary.

## Architecture

### Data Flow

```
User completes conversation (status -> completed)
  -> Application enqueues "conversation-analytics" job via pg-boss
  -> Worker fetches all messages for the conversation
  -> Worker sends single Bedrock LLM call with structured prompt
  -> LLM returns JSON: { sentiment, sentimentScores, isResolved, confidenceScore, emotionalTone, summary, firstUserQuery, language }
  -> Worker stores result in conversation_analytics table
  -> Dashboard queries PostgreSQL with tenant-scoped filters and aggregations
```

### Tech Stack (existing)

- Database: PostgreSQL + Prisma
- Workers: pg-boss (boss/executor pattern)
- LLM: Amazon Bedrock via @ai-sdk/amazon-bedrock
- Frontend: Next.js 15 App Router, shadcn/ui, Recharts
- State: TanStack React Query for server state

## Worker Job: conversation-analytics

### Registration

File: `apps/workers/src/jobs/conversation-analytics/register.ts`

Registers with pg-boss:
- Queue name: `conversation-analytics`
- Retries: 3
- Retry delay: 30s
- Expiry: 10 minutes

### Handler

File: `apps/workers/src/jobs/conversation-analytics/handler.ts`

**Input payload:**
```typescript
interface ConversationAnalyticsPayload {
  conversationId: string;
  tenantId: string;
}
```

**Steps:**
1. Fetch conversation + all messages from DB
2. Skip if analytics record already exists (idempotent)
3. Concatenate messages into a transcript string
4. Call Bedrock LLM with structured prompt
5. Parse JSON response
6. Insert ConversationAnalytics record

### LLM Prompt

```
You are a conversation analyst. Analyze the following chatbot conversation and return a JSON object with these fields:

- sentiment: The overall sentiment of the user's messages. One of: "POSITIVE", "NEGATIVE", "NEUTRAL", "MIXED"
- sentimentScores: Confidence scores for each sentiment as decimals (0-1). Object with keys: positive, negative, neutral, mixed. Must sum to ~1.0.
- isResolved: Whether the user's query/issue was resolved by the end of the conversation. Boolean.
- confidenceScore: How confident you are in the resolution assessment (0.0 to 1.0).
- emotionalTone: The user's emotional state. Object with keys: happy, frustrated, neutral. Each 0.0-1.0, should sum to ~1.0.
- summary: A 1-3 sentence summary of what the conversation was about and the outcome.
- firstUserQuery: The user's first meaningful question or request (exclude greetings like "hi", "hello"). Max 200 chars.
- language: The primary language the user communicated in (ISO 639-1 code, e.g., "en", "hi", "ta").

Return ONLY valid JSON, no markdown, no explanation.

CONVERSATION:
{transcript}
```

### Triggering

Enqueued when a conversation's status changes to `completed`. Add a pg-boss publish call in the conversation completion flow.

## API Routes

All routes are tenant-scoped via the existing middleware (x-tenant-id header).

### GET /api/analytics/summary

Returns aggregated analytics for the dashboard.

**Query params:** `fromDate`, `toDate`, `sentiment`, `resolvedStatus`

**Response:**
```typescript
{
  counts: {
    total: number;
    active: number;
    completed: number;
    resolved: number;
    unresolved: number;
    analyzed: number;
  };
  sentimentDistribution: {
    positive: number;
    negative: number;
    neutral: number;
    mixed: number;
  };
  resolutionRate: number;
  avgConfidence: number;
  trends: {
    sentiment: Array<{ date: string; positive: number; negative: number; neutral: number; mixed: number }>;
    resolution: Array<{ date: string; resolved: number; unresolved: number }>;
    volume: Array<{ date: string; count: number }>;
  };
  topQueries: {
    mostAsked: Array<{ query: string; count: number }>;
    positive: Array<{ query: string; count: number }>;
    negative: Array<{ query: string; count: number }>;
  };
  nps: {
    score: number;
    promoters: number;
    passives: number;
    detractors: number;
    totalWithRatings: number;
    ratingDistribution: Record<number, number>;
  };
}
```

### GET /api/analytics/sessions

Returns paginated list of conversations with their analytics data.

**Query params:** `fromDate`, `toDate`, `search`, `sentiment`, `resolvedStatus`, `status`, `page`, `limit`

**Response:** sessions array with analytics fields + pagination metadata.

### GET /api/analytics/sessions/[id]

Returns full session detail including messages and analytics.

**Response:** session metadata, messages array, analytics object.

## Frontend Pages

### Route Structure

```
apps/web-ui/app/(dashboard)/
  analytics/
    page.tsx                -> Analytics dashboard (charts)
    sessions/
      page.tsx              -> Sessions list
      [id]/
        page.tsx            -> Session detail
```

### Analytics Dashboard Page (/analytics)

**Sections:**
1. KPI Cards (6 cards): Total Conversations, Active Sessions, Completed Sessions, Resolution Rate, Positive Sentiment %, Total Messages
2. Sentiment Overview: Donut chart + stacked area chart (trends)
3. Resolution Overview: Pie chart + area chart (trends)
4. Top User Queries: 4-column grid (Most Asked, Positive, Negative, Mixed)
5. NPS Section: Gauge + Rating distribution + Promoters vs Detractors

### Sessions Page (/analytics/sessions)

**Layout:**
1. Filter bar (date range, search, sentiment, resolution status)
2. Tab bar (All | Active | Completed with counts)
3. Session table with columns: ID, first query, status, sentiment, resolution, messages, rating, date
4. Pagination

### Session Detail Page (/analytics/sessions/[id])

**Two-column layout:**
- Left (60%): Session metadata card + User feedback card + AI Analysis card (tabs: Summary, Sentiment)
- Right (40%): Chat transcript (scrollable chat bubbles)

## Error Handling

- Worker failures: pg-boss retries 3 times. Graceful degradation if exhausted.
- LLM parse failures: Log error, store record with null sentiment.
- Empty conversations: Skip analysis for 0 user messages.
