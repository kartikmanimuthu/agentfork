## What changed

Graph agents can now pause at Human nodes and resume when the user replies. The full flow:

1. **Pause**: When a Human node executes, it sets `__paused: true` in state channels. The `GraphExecutor` detects this, calls the `onPause` callback, and stops the loop. The inference route's `onPause` handler persists state to `PausedExecution` (DB) and marks `ApiKeyExecution.status = 'paused'`.

2. **Resume**: Client calls `POST /api/v1/resume` with a `resumeToken`. The endpoint atomically claims the token (CAS — `updateMany WHERE resumedAt IS NULL AND expiresAt > now`), enqueues a `resume-agent-execution` pg-boss job, and returns 202. The worker restores graph state, injects the user's reply into the output channel, clears `__paused`, and runs `executeFromState()`.

3. **Polling**: Client polls `GET /api/v1/executions/:id` for completion status.

4. **Expiry**: A nightly cron at 00:30 UTC (06:00 IST) sweeps expired paused executions and marks parent `ApiKeyExecution` rows as `failed`.

Key implementation details:
- `GraphExecutor` was refactored: the while-loop extracted into private `runLoop()`, both `execute()` and `executeFromState()` call it (DRY).
- `executeFromState()` accepts a saved `GraphState` and resumes from `currentNodeId` — no re-running prior nodes.
- Token claiming is atomic via CAS (`claimToken()`). Two concurrent resume requests on the same token — only one succeeds.
- The graph agent stub in the inference route was replaced with real `GraphExecutor` execution with SSE streaming.
- Chained human nodes are supported: the worker's `onPause` handler creates a new `PausedExecution` for the second pause.

## Why

Graph agents with Human nodes need to pause execution and wait for user input — potentially for hours or days. Three design problems had to be solved:

1. **Serverless limits**: Vercel (60s) and Lambda+API Gateway (29s) kill long-running connections. Sync resume is impossible on serverless. Async resume (202 + worker job + polling) works everywhere.

2. **Double-resume race**: If two requests hit `/resume` with the same token simultaneously, both could trigger execution. Atomic CAS token claiming (`updateMany WHERE resumedAt IS NULL`) ensures only one succeeds.

3. **DRY violation**: Naive `executeFromState()` would duplicate the entire execution loop. Extracting `runLoop()` as a private method keeps pause detection in one place.

## Research

- **LangGraph**: Uses durable state checkpointer per node, `interrupt()` primitive, single-use `thread_id` for resume. We match on durable state and single-use tokens but skip per-node checkpointing (acceptable for chatbot use case).
- **AWS Step Functions**: Task tokens for resume, `SendTaskSuccess/Failure` API, state machine manages transitions. Our pattern is analogous — `PausedExecution` row is the task token.
- **Temporal**: Workflow replay with determinism cache, activity timeouts, signal-based resume. Our approach is simpler — no replay, no workflow DSL, just a while-loop with state serialization.
- **Postgres `LISTEN/NOTIFY`**: Noted as a follow-on upgrade for streaming on resume (no Redis needed). Drop-in once polling is proven.

## Watch for

- `resumeToken` is generated by `HumanNodeExecutor` (via `crypto.randomUUID()`) and stored in `PausedExecution.resumeToken` — NOT auto-generated by Prisma's `@default(uuid())`. The SSE event carries the token from the executor, through `PauseInfo`, to the DB. If `create()` doesn't receive the token, a mismatch occurs and resume is permanently broken.
- `wasPaused` flag in `runLoop()` prevents `execution_complete` from being emitted on paused runs. If removed, clients receive both `execution_paused` AND `execution_complete`, which is confusing.
- Graph agent quota tracking currently passes `0` tokens to `quotaService.incrementUsage()`. LLM token usage inside graph execution is not aggregated — follow-up needed.
- Expiry cron uses `take: 500` limit when marking parent executions. At very high scale, some expired executions may not be swept in a single run.
- `ApiKeyExecution` is referenced by `PausedExecution.executionId` as loose coupling (no FK). If an execution is hard-deleted, the resume job will find a null execution and return early with a warning log.
