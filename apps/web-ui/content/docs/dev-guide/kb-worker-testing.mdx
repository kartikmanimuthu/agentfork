## Overview

Knowledge base sync is driven by a **pg-boss** job queue. The flow:

1. **Frontend** — user uploads a file or clicks the play icon on a URL-type source
2. **API** — `POST /api/knowledge-bases/{id}/upload` creates a document record and returns a pre-signed S3 URL; `POST /api/knowledge-bases/{id}/ingest` enqueues a `document-ingestion` job after the client uploads to S3
3. **API** — `POST /api/knowledge-bases/{id}/sources/{sourceId}/sync` enqueues a `web-crawl` job
4. **Worker** — `handleWebCrawl` crawls the URL, uploads text to S3, creates `Document` records, then enqueues `document-ingestion` jobs
5. **Worker (ingestion)** — `handleDocumentIngestion` parses, chunks, embeds, and stores searchable vectors

## Local Testing Script

A standalone script in `apps/workers/src/scripts/test-kb-sync.ts` exercises the entire pipeline directly without needing the frontend running.

### Commands

```bash
cd apps/workers

# 1. Test the web crawler in isolation (no DB writes)
npx tsx --env-file=../../.env src/scripts/test-kb-sync.ts \
  --crawl-only --url https://example.com --max-pages 1

# 2. Test sync for an existing source
npx tsx --env-file=../../.env src/scripts/test-kb-sync.ts \
  --kb-id <kb-id> --source-id <source-id> --tenant-id <tenant-id>

# 3. Full pipeline — auto-create test records, run sync + ingestion, then clean up
npx tsx --env-file=../../.env src/scripts/test-kb-sync.ts \
  --create --full-pipeline --cleanup

# Or use the package.json alias
bun run test:kb-sync -- --create --cleanup
```

### Script Options

| Flag | Description |
|------|-------------|
| `--create` | Auto-create a test `KnowledgeBase` + URL `DataSource` |
| `--kb-id` | Existing KnowledgeBase ID |
| `--source-id` | Existing DataSource ID |
| `--tenant-id` | Tenant ID (defaults to first active tenant when `--create`) |
| `--full-pipeline` | Also run `document-ingestion` (chunking + embeddings) after sync |
| `--cleanup` | Delete created test records after the run |
| `--crawl-only` | Test the web crawler without any database writes |
| `--url` | Target URL for `--crawl-only` |
| `--crawl-depth` | Link hops to follow (default: `0`) |
| `--max-pages` | Pages to crawl (default: `10`) |

### What the Script Validates

- **Sync phase** — data source status transitions from `syncing` to `active`, `lastSyncAt` is set, and `Document` records are created
- **Ingestion phase** (with `--full-pipeline`) — documents reach `READY` status, chunks are stored, and embeddings are generated via Bedrock
- **Cleanup phase** (with `--cleanup`) — removes test documents, chunks, data source, and knowledge base

> **Note:** The script reports `Chunks with embeddings: 0/N` even when embeddings are written successfully. This is expected — Prisma cannot read back `Unsupported("vector")` columns through the ORM. The `READY` document status confirms embeddings were stored correctly via raw SQL.

## Running the Worker Server

To process jobs via the actual worker daemon:

```bash
# Terminal 1 — start the worker
bun run dev:workers

# Terminal 2 — trigger a sync via curl (or use the UI)
curl -X POST \
  http://localhost:3005/api/knowledge-bases/{kb-id}/sources/{source-id}/sync
```

The worker polls pg-boss every few seconds and picks up jobs automatically.

## Environment Setup

The following `.env` variables are required for the full pipeline:

```bash
AWS_PROFILE=STX-CLOUD-PLATFORM   # AWS profile with Bedrock + S3 access
AWS_REGION=us-east-1             # Must match the S3 bucket region
S3_BUCKET=chatbot-970547372609-us-east-1
```

For local MinIO (optional, file upload only — Bedrock still requires real AWS):

```bash
S3_ENDPOINT=http://localhost:9000
S3_FORCE_PATH_STYLE=true
S3_ACCESS_KEY_ID=minioadmin
S3_SECRET_ACCESS_KEY=minioadmin
```

## Common Issues

| Symptom | Cause | Fix |
|---------|-------|-----|
| `jobId: null` when enqueuing | pg-boss v10 requires queues to exist before `send()` | Fixed — `createQueue()` is called before `work()` in all register files |
| `Failed to upload file to storage` | CORS blocked browser PUT to S3/MinIO, or wrong AWS credentials signing the URL | Set `S3_ACCESS_KEY_ID`/`S3_SECRET_ACCESS_KEY` for MinIO, or ensure `AWS_PROFILE` is set for real S3 |
| `301 Moved Permanently` on S3 upload | `AWS_REGION` doesn't match the bucket's actual region | Set `AWS_REGION=us-east-1` (bucket `chatbot-970547372609-us-east-1` is in us-east-1) |
| `The security token included in the request is invalid` | `AWS_ACCESS_KEY_ID` in `.env` overrides the AWS profile | Remove `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` from `.env`; use `AWS_PROFILE` instead |
| `column "embedding" does not exist` | Schema not pushed after adding vector column | Run `bunx prisma db push` from project root |
| `column "search_text" does not exist` | tsvector column missing from DB | Run `bunx prisma db push` or `ALTER TABLE document_chunks ADD COLUMN IF NOT EXISTS search_text tsvector` |
| `Cannot find package 'pdf-parse'` | Parser packages not installed at workspace root | Run `bun add pdf-parse@1.1.1 mammoth xlsx` at project root |
| `pdfParse is not a function` | `pdf-parse` v2 changed its API | Use `pdf-parse@1.1.1` (v1 exports a callable function) |
| `Queue web-crawl not found` | pg-boss v10 doesn't auto-create queues | Fixed — `createQueue()` added to all register files |
| "Crawl complete, pages: 0" | `excludePatterns` was an empty array `[]`; the crawler treated it as "exclude all" | Fixed in `libs/knowledge-base/src/web-crawler/index.ts` — check `excludePatterns.length > 0` before filtering |
| `AWS credential provider failed` | No AWS credentials available | Set `AWS_PROFILE` in `.env` to a profile with Bedrock access |

## Files to Know

|------|---------|
| `apps/workers/src/scripts/test-kb-sync.ts` | Local testing script |

| `apps/web-ui/app/api/knowledge-bases/[id]/sources/[sourceId]/sync/route.ts` | API route that enqueues the sync job |
